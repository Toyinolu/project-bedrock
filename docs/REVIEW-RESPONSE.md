# Review Response — Project Bedrock

Most earlier review comments were recorded **while the environment was temporarily
torn down to avoid AWS charges** (EKS has no free tier). When the cluster is down,
a reviewer sees "app not reachable / unable to connect to cluster / no lambda /
no logging / dbs not used" — none of which reflect the actual implementation.

**The environment is now live and every requirement is verifiable.** Below, each
review comment is mapped to concrete, reproducible evidence.

> **Live access details are in the submitted Google Doc** (app URL + bedrock-dev-view
> credentials). They are intentionally NOT committed to this public repo — committing
> live AWS keys trips AWS secret-scanning and quarantines the key.

## Per-comment response

### "Unable to connect to cluster" / "deployment likely not up" / "App not reachable"
**Resolved — was offline during review.** The cluster is `ACTIVE`, all 7 application
pods are `Running`, and the store returns HTTP 200 at the ALB URL in the Google Doc.
The `bedrock-dev-view` credentials produce a working kubeconfig:
```bash
aws eks update-kubeconfig --name project-bedrock-cluster --region us-east-1
kubectl get pods -n retail-app      # lists 7 running pods

```

### "in-cluster db used" / "dynamo resource not used" / "Only a MySQL rds instance was used" / "Utilization of RDS and dynamo could not be validated"
**Not accurate — managed services are used and wired into the app:**
- **catalog → Amazon RDS MySQL** (`project-bedrock-mysql`). Pod env:
  `RETAIL_CATALOG_PERSISTENCE_ENDPOINT=project-bedrock-mysql....rds.amazonaws.com:3306`
- **orders → Amazon RDS PostgreSQL** (`project-bedrock-postgres`). Pod env:
  `RETAIL_ORDERS_PERSISTENCE_ENDPOINT=project-bedrock-postgres....rds.amazonaws.com:5432`
- **carts → Amazon DynamoDB** (`retail-store-carts`) via **IRSA**
  (`project-bedrock-cart-service-role`), `RETAIL_CART_PERSISTENCE_PROVIDER=dynamodb`.
- There are **no in-cluster mysql/postgres pods** — the chart's in-cluster DBs are
  disabled (`mysql.create=false`, `postgresql.create=false` in `helm/values-override.yaml`).

Verify:
```bash
aws rds describe-db-instances --query "DBInstances[?starts_with(DBInstanceIdentifier,'project-bedrock')].[DBInstanceIdentifier,Engine]"
kubectl exec -n retail-app deploy/catalog -- env | grep RETAIL_CATALOG_PERSISTENCE_ENDPOINT
kubectl exec -n retail-app deploy/orders  -- env | grep RETAIL_ORDERS_PERSISTENCE_ENDPOINT
kubectl get pods -n retail-app | grep -iE 'mysql|postgres'   # (none)
```

**Resolved.** The `bedrock-dev-view` IAM user has `ReadOnlyAccess` (managed) +

### "Hardcoded credentials" / "service dependency credentials incorrectly referenced"
**Not accurate.** No credentials are hardcoded in any committed file. DB passwords are
generated by Terraform (`random_password`), stored in **AWS Secrets Manager**, and
injected as **Kubernetes Secrets** referenced by name (`secret.create: false`,
`passwordKey: ...` in `helm/values-override.yaml`). Git history contains no real keys
(verified). `grading.json` has the three credential fields redacted on purpose.

### "CICD setup does not meet requirement" / "CICD pipeline unsuccessful" / "Incorrect event driven CICD pipeline" / "Pipeline did not include automated app deployment"
**Fixed and reproducible:**
- **PR → `terraform plan`, posted as a PR comment**; **merge to main → `terraform apply`**
  (`.github/workflows/terraform-plan.yml`, `terraform-apply.yml`). Triggers are
  `on: pull_request` and `on: push: branches:[main]` — exactly the spec. (Earlier the
  workflows had a `paths: [terraform/**]` filter that suppressed runs on non-Terraform
  PRs; that filter has been removed so plan runs on **every** PR and apply on **every**
  merge.)
- **Automated app deployment:** `terraform apply` itself deploys the app —
  `module.k8s.helm_release.retail_store` installs the committed chart
  (`helm/chart/`) with the RDS/DynamoDB values override. No manual deploy step.
- **From-scratch reliability:** the apply workflow now runs a **two-phase apply**
  (AWS infra → poll EKS `/readyz` until reachable → in-cluster k8s/Helm resources),
  which eliminates the EKS-endpoint DNS-propagation race that can fail a single-pass
  apply. The Apply run on the latest commit is **green end-to-end**.
- **Security:** OIDC auth (`secrets.AWS_ROLE_ARN`), no hardcoded keys in workflows.

### "Wrong app deployed"
